# TIMER.md — フレームペーシング（メイン CPU の周期タイマ IRQ を数える deadline 方式）

メインループの周期を一定に保つ「フレームペーシング」を、**メイン CPU が持つ約 2ms 周期のタイマ IRQ を IRQ ハンドラで数える「経過 tick カウンタ」**で実現します。`timer_start()` でカウンタを 0 に戻し、`timer_get()` が一定値（`FRAME_TARGET`）に達するまでロックする **deadline 方式**です。実装は [src/asm_timer.s](../src/asm_timer.s) と [src/c_main.c](../src/c_main.c)。

> ⚠️ IRQ ベクタ設置・タイマ IRQ の挙動は機種依存を含み、**実機/エミュレータでの検証が前提**です。値が未確定でも `timer_get()` が増えないだけで、待ち側の安全上限で degrade（= 速く走るだけ）しハングはしません（§4・§7）。

## 1. なぜ deadline 方式か

- 初代 FM-7 は **メイン CPU から VSYNC（垂直帰線）を直接検出できません**（`$D430` の表示 status や `$FD12` は FM77AV 以降のみ有効）。
- CPU サイクル数に依存する固定ウェイト（busy-wait）では「処理の重い/軽い」で 1 周期がばらつきます（移動中の重い `MOVE_SPRITE` フレームと停止フレームで速度が変わる）。
- そこで「**フレーム先頭で経過時間カウンタを 0 に戻し、 処理が終わったら一定時間が経つまでロックする**」 deadline 方式にします。処理が重くても軽くても 1 フレームの実時間が一定（`FRAME_TARGET × 約2ms`）に揃います。

## 2. 時間源 — メイン CPU の周期タイマ IRQ（約 2ms）

FM-7 のメイン 6809 には **約 2ms 周期（≈491Hz）の周期タイマ IRQ** があります。

| ポート | 役割 |
| --- | --- |
| `$FD02` (W) | IRQ 許可（bit2=タイマ / bit0=キーボード / bit1=プリンタ。 1=許可） |
| `$FD03` (R) | IRQ 要因フラグ（active-low: 0=発生。 bit2=タイマ / bit0=キーボード） |

`$FD02` bit2 を立てるとタイマ IRQ が許可され、約 2ms ごとに IRQ が発生します。IRQ ハンドラが 16-bit カウンタ `frame_tick` を ++ します（1 tick ≈ 2ms）。

## 3. なぜ「IRQ で数える」のか（ポーリング/FIRQ ではなく）

- **なぜ数えるのか（ポーリングではダメな理由）**: 単一のステータスフラグをポーリングすると、処理（描画など）の最中に複数の周期が過ぎても「フラグが立っている」 に飽和するだけで、**何 tick 過ぎたか分かりません**。それでは「一定時間まで待つ」 deadline が組めません。IRQ で数えれば**処理の最中に過ぎた tick も取りこぼさず**カウントできます。
- **なぜ IRQ で、FIRQ ではないのか**: FM-7 の BREAK キーは **FIRQ 要因**（`$FD04` bit1）で、押下中ずっとレベルでアサートし続けます。FIRQ を許可すると BREAK 押下中ずっと FIRQ が再発火し、メインループが餓死します（= 実際に発生した）。メインの周期タイマは **IRQ 要因**（`$FD02`/`$FD03` bit2）なので、**IRQ だけ許可し FIRQ はマスクのまま**にすれば、BREAK と干渉しません（BREAK は従来どおり `break_check()` が `$FD04` bit1 をポーリング）。

`timer_init()` は IRQ ベクタ **`$FFF8`**（IPL が `$FD0F` write で RAM モードにして以降 RAM）に自前ハンドラを設置し、`$FD02` を `$05`（タイマ bit2 + キーボード bit0、 §6 参照）にして、`ANDCC #$EF` で I フラグだけ解除（FIRQ はマスクのまま）します。

## 4. メインループでの使い方

```c
timer_init();    /* 起動時 1 回: IRQ ベクタ設置 + タイマ/キーボード IRQ 許可 + IRQ 解禁 */
timer_start();   /* 経過カウンタを 0 に */
for (;;) {
    /* 前フレーム開始から FRAME_TARGET 個ぶんの tick (= 約2ms 単位) が
     * 溜まるまでロック。 ロック + リセットはループ先頭に置く (= 内部に
     * continue があるため全周回を均一にペーシング)。 */
    { unsigned g = 0; while (timer_get() < FRAME_TARGET && g < PACE_SAFETY_CAP) g++; }
    timer_start();
    /* ... キー入力 / 移動 / ボール更新 / 描画 / サウンド ... */
}
```

- `FRAME_TARGET` … 1 フレームに使う tick 数。**1 tick ≈ 2ms** なので、`16` で約 32ms/フレーム（≈30fps）。大きいほど 1 フレームが長く、全体がゆっくりになります。処理がこれより重いフレームは待ちが入らず、その分だけ degrade します（= タイマ基準なので速くなりすぎはしない）。
- `PACE_SAFETY_CAP` … タイマ IRQ が来ない環境で `while` が無限ループにならないための上限。`timer_get()` が増えないと頭打ちにして degrade します。

`timer_get()` の待ちに費やしたスピン回数を `SCORE` 欄にデバッグ表示します（安定した値ならタイマ IRQ 順調、`PACE_SAFETY_CAP` 付近に張り付くならタイマ IRQ が来ていないサイン）。

## 5. C/asm の関数

| 関数 | 実体 | 役割 |
| --- | --- | --- |
| `timer_init()` | asm_timer.s | IRQ ベクタ `$FFF8` 設置 + `$FD02=$05` でタイマ/キーボード IRQ 許可 + I フラグ解除 |
| `timer_start()` | asm_timer.s | 経過カウンタ `frame_tick` を 0 に |
| `timer_get()` | asm_timer.s | `timer_start()` からの経過 tick（16bit、1 tick ≈ 2ms）を返す |
| IRQ ハンドラ | asm_timer.s | `$FD03` を読み、タイマ(bit2)なら `frame_tick++`、キーボード(bit0)なら §6 |

## 6. キーボードも同じ IRQ ハンドラで（IRQ 駆動）

FM-7 のキーボードは **キーボード IRQ（`$FD02` bit0）を許可しないとデータが来ません**。一方、CPU IRQ を許可するとキーボード IRQ も発生し、その IRQ フラグは **`$FD01`（キーコード）を読むまでクリアされない**ため、放置すると再発火して餓死します。

そこで **キーボードも IRQ 駆動**にします。同じ IRQ ハンドラが、キーボード IRQ（`$FD03` bit0=0）のとき `$FD01` を読み（= キーボード IRQ を ack）、キーコードを共有バッファ `_kbd_buf`（[asm_kbd.s](../src/asm_kbd.s)）に格納します。`key_check()` はこのバッファを取り出して返すだけです（取り出しの間だけ IRQ をマスクして read+clear を atomic に）。

> タイマ IRQ のフラグは `$FD03` を読むことで ack されますが、キーボード IRQ のフラグは `$FD01` を読まないと ack されません。ハンドラはこの両方を 1 回の IRQ で処理します。

## 7. エミュ/実機での検証ポイント（★）

1. **`$FFF8` の RAM 設置**：RAM モードで IRQ ベクタが書けていて、タイマ IRQ で自前ハンドラに入るか（`timer_get()` が増えるか）。`SCORE` のスピン値が `PACE_SAFETY_CAP` に張り付くならここを疑う。
2. **タイマ IRQ フラグの ack**：`$FD03` の read でタイマフラグがクリアされる前提です（クリアされないと再発火して激遅/フリーズ）。
3. **キーボード IRQ**：`$FD01` の read で ack される前提（§6）。キーが効かない/押すと固まる場合はここ。
4. **周期**：タイマ IRQ の周期（約 2ms）が機種で揃っているか。フレームを遅くしたいときは `FRAME_TARGET` を増やします。
