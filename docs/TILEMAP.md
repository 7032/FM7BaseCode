# TILEMAP.md — タイルマップ背景 設計メモ(提案)

このドキュメントは「タイルマップ背景」を実装するための **設計提案** です。
まだ実装はしていません。現状のテンプレートを正確に踏まえたうえで、データ構造・
描画コマンド・既存処理との整合・スクロール方式・実装手順を提案としてまとめます。

---

## 0. 現状の確認(出発点)

実装に入る前に、いま背景がどう描かれているかを押さえておきます。

- 背景は **`backimage.png`(64x64 モノクロ)を全画面にタイル状に敷いている**。
  - `scripts/bgtile_to_asm.py` が 64x64 を B plane 512 byte(64 line × 8 byte)に変換。
  - 起動時に `sub_load_bgtile()` で sub の `$CB00` へ転送し、`asm_subprog.s` の
    `DRAW_BG` が「R/G 全クリア + B plane に同じ 64x64 タイルを横64px/縦64line 周期で
    全画面に敷く」。 つまり **全画面が同じ 1 枚のタイル**(セルごとの絵柄は持たない)。
- スプライト/ボール移動時の背景消去は、旧位置の **R/G plane を 0 クリアするだけ**。
  - B plane(= 背景タイル)には触らないので、消した跡に自然と背景が戻る。
- VRAM は **B/R/G の 3 プレーン**だが、本テンプレは **背景=B / 前景=R・G** と分担
  (詳細は [SPRITE.md](SPRITE.md) / [SUBPROGRAM.md](SUBPROGRAM.md))。1 ライン = 80 byte、200 ライン。
- サブ側メモリ配置(触ってよい範囲):
  - コンソール RAM `$C000-$CFFF`
    - subprog コード `$C300-$C6FF`
    - スプライト `$C700-$CAFF`
    - 作業変数 `$C200-`
  - `$D000` 以降は触らない。
- サブへの転送は TEST コマンド(CMD `$3F`)経由の `sub_takeover`。
  描画は自前 subprog のコマンドを `sub_call` で発行する。

**単一タイルの敷き詰めは既に実装済み**。 ここから先のタイルマップ化とは、この
**「全画面に同一タイルを敷く」を「セルごとにタイル番号を持つマップで、場所ごとに
違うタイルを敷く」へ拡張する** ことです。

---

## 1. 目的

現状の「全画面に同一タイル」の代わりに、セルごとに任意のタイル(例 8x8 または
16x16 ドット)を並べた背景(タイルマップ)を表示・スクロールできるようにする。狙いは次のとおり。

- 背景を「絵」として自由に作れる(地形・床・壁・装飾など)。
- ステージごとに見た目を差し替えられる(マップデータを入れ替えるだけ)。
- 将来のスクロール(横/縦移動するゲーム画面)への足場にする。

現状は背景タイルを差し替えれば模様は変わるが「全画面同一」止まり。タイルマップなら
**マップデータ(セルごとのタイル番号)を差し替えるだけ** で場所ごとに違う絵柄の
背景を作れる、という性質を得るのが本質。

---

## 2. データ設計

タイルマップは大きく 2 つのデータからなる。

### 2-1. タイルパターン集(タイルの絵)

各タイルの B/R/G ビットマップ。タイル 1 枚を 8x8 とした場合の見積もり:

- 8 ドット幅 = 1 byte/ライン。8 ライン = 8 byte/プレーン。
- 3 プレーン(B/R/G)なので **8 x 3 = 24 byte/タイル**。
- 16x16 にすると 1 ライン 2 byte x 16 ライン x 3 プレーン = **96 byte/タイル**。

タイル枚数ごとの概算(8x8 の場合):

| タイル枚数 | 必要 byte (24 byte/枚) |
|-----------|------------------------|
| 16        | 384 byte               |
| 32        | 768 byte               |
| 64        | 1536 byte              |

### 2-2. タイル番号マップ(どこに何を置くか)

画面を cell 単位に区切り、各 cell に「どのタイルか」を 1 byte で書く。

- 8x8 タイルで全画面を埋める場合（本テンプレの表示は 1 ライン 80 byte = 横 640
  ドット × 200 ライン、§0）: 横 640/8 = 80 cell、縦 200/8 = 25 cell。
  - マップ = 80 x 25 = **2000 byte**(1 byte/cell)。
- 16x16 タイルなら 40 x 13(端数切上)= **520 byte** 程度。

> 横ドット数は表示モードに依存する（本テンプレは 1 ライン 80 byte = 640 ドット）。
> 別モードに変える場合は実際の設定に合わせて再計算すること。
> ここでは「1 byte/cell でマップを持つ」という構造だけが要点。

### 2-3. サブ RAM のどこに置くか

現状の空きは実質 `$CF00-$CFFF`(256 byte)程度しかない。これは小さいので、
タイル数・マップサイズ次第で配置を再計算する必要がある。

- **小規模(タイル数少・マップ小)**: パターン集とマップを `$CF00` 付近の空きや
  作業変数領域の調整で収める。
- **中〜大規模**: サブ RAM に全部は載らない。次のいずれかを採る。
  - パターン集だけサブに常駐させ、マップは画面更新時にメインから少しずつ転送する
    (動的ロード)。
  - マップを圧縮(連長圧縮など)してサブで展開する。
  - 1 画面分ずつメインから `sub_call` で流し込み、サブ側はバッファとして使う。

> `$D000` 以降は触らない、という制約は維持する。容量が足りなければ
> 「サブに常駐するのはパターン集」「マップは流し込み」が現実的な落としどころ。

## 3. 描画コマンド設計(subprog cmd 案)

既存の subprog は `sub_call` でコマンド番号を渡して描画を依頼する形。
タイルマップ用に新コマンドを追加する案。

- **`DRAW_TILE`**: 1 cell 分を描く。
  - 引数: cell の位置(列・行、または直接 VRAM オフセット)+ タイル番号。
  - 処理: タイル番号からパターン集の先頭を引き、B/R/G 各プレーンへ 8(or 16)
    ライン分転送する。
- **`DRAW_TILEMAP`**: マップ全体を一括描画。
  - 引数: マップ先頭アドレス(または常駐マップを使う)。
  - 処理: cell を順に走査して内部的に `DRAW_TILE` 相当を繰り返す。
    初期描画・画面切替時に使う。

### 既存処理の流用ポイント

- `PUT_CELL` / `DRAW_CHAR` は「指定 cell に固定サイズのビットマップを B/R/G へ
  書く」処理であり、タイル描画とやることがほぼ同じ。
  - cell → VRAM オフセット変換、プレーンごとのライン転送ループはそのまま転用できる。
  - 「文字フォントを引く」部分を「タイル番号でパターン集を引く」に差し替える発想。
- スプライト描画(`$C700-$CAFF`)のビットマップ転送ルーチンも、
  プレーン分割の書き込み方として参考になる。

## 4. 背景消去との整合(重要)

現状はスプライトが動くと `bg_fill_box` が旧位置の **R/G plane を 0 クリアするだけ**で、
背景(B plane のタイル)には触らない。だから消した跡には B plane に残っている背景が
そのまま戻る(§0)。

ここが本テンプレの利点で、**タイルマップ背景を B plane だけで持つ限り、この消去
(R/G クリア)はそのまま使える**。スプライトを消すと、その場所の B plane に元から
描いてあるタイルマップ背景が自動的に見えるので、cell ごとの描き直しは不要。

注意が要るのは次の場合:

- **色つきタイル(R/G を使う背景)を持ちたい**とき。背景が R/G を使うと前景(R/G)と
  衝突し、スプライトを消す R/G クリアで背景まで消えてしまう。この場合は
  「スプライト矩形に重なる cell のタイルを `DRAW_TILE` で描き直す」消去へ
  切り替える(端をまたぐ cell も対象にする)。
- スクロールで背景全体を動かすとき(§5)は、いずれにせよ背景を描き直す。

> まとめ: **背景を B plane のみ・前景を R/G に保つ**なら、消去ルーチンは現状の
> R/G クリアのままでよい(タイルマップ化と消去の差し替えはセットではない)。
> 色つき背景やスクロールを入れる段になって初めて消去方式を見直す。

## 5. スクロール方式の比較

### 5-1. ハードスクロール(オフセットレジスタ)

FM-7 のディスプレイオフセット(表示開始アドレス)を書き換えて画面全体をずらす。

- 利点: VRAM を描き直さずに画面が動く → **高速**。
- 注意:
  - **VRAM 全体が一様に動く**(背景もスプライトも一緒にずれる)。
    スプライト側はオフセットを考慮した座標で描き直す必要がある。
  - 書き換えは **帰線期間(VBLANK)に同期** させないと表示が乱れる。
  - 1 画面分動いたら、見えてくる端に新しいタイル列/行を補充して描く
    (継ぎ足し描画)。
- 向き: 連続スクロールするゲーム画面。

### 5-2. ソフトスクロール(描き直し)

毎フレーム、ずらした位置でタイルマップを描き直す。

- 利点: 実装が素直。オフセットの同期問題に悩まされない。
- 欠点: 毎回 VRAM を塗り直すので **重い**。広い画面の常時スクロールには不利。
- 向き: たまにしか動かない/部分的なスクロール、まず動かして確認したい段階。

> 最初はソフトスクロールで「タイルが正しく並ぶ」ことを確認し、
> 速度が要るところでハードスクロールへ移行するのが安全。

## 6. 実装ステップ(提案)

1. **データ変換スクリプト**
   - 画像/定義ファイル → タイルパターン集(B/R/G 形式)+ タイル番号マップへ
     変換するツールを用意する(ホスト側、ビルド時生成)。
   - 出力はサブへ転送しやすいバイナリ/include 形式にする。
2. **subprog cmd 追加**(`asm_subprog.s`)
   - `DRAW_TILE` / `DRAW_TILEMAP` を実装。`PUT_CELL` / `DRAW_CHAR` の
     cell→VRAM 変換とプレーン転送を流用する。
3. **背景消去**(`asm_subprog.s`) — 背景を B plane のみで持つなら現状の `bg_fill_box`
   (R/G クリア)のままでよい(§4)。色つきタイルを使う場合のみ「該当 cell のタイル
   再描画」へ差し替える。
4. **C API**
   - 新コマンドを `sub_call` で発行するラッパ関数を用意
     (例: `draw_tile(col, row, id)` / `draw_tilemap()`)。
5. **c_main 連携**
   - 起動時の背景生成を `DRAW_BG`(全画面同一タイル)から `draw_tilemap()` へ差し替え。
   - スプライト更新ループの消去呼び出しを新方式に合わせる。
6. **(任意)スクロール対応**
   - まずソフトスクロールで検証 → 必要ならオフセットレジスタ + VBLANK 同期 +
     端の継ぎ足し描画を追加。

### 手を入れる既存ファイル(目安)

- `asm_subprog.s`: 現状の `DRAW_BG`(単一タイル敷き)を拡張 or `DRAW_TILEMAP` を新設。
  (消去 `bg_fill_box` は背景=B/前景=R・G を保つ限り R/G クリアのままで可。 §4 参照。)
- C 側(描画 API を持つソース): `sub_draw_tilemap` / `sub_load_tilemap` ラッパ追加。
- `c_main`(メインループ): 背景初期化を `DRAW_BG` から `DRAW_TILEMAP` へ差し替え。
- ビルド構成: タイルパターン集 + マップ配列の変換スクリプト(`scripts/`)と生成物の組み込み。
