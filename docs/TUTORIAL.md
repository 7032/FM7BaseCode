## 0. はじめに — 完成形と読み方

### このテンプレで動くもの（完成形）

ビルドしてディスクから起動すると、 こうなります:

- 画面いっぱいに **`assets/backimage.png`（64×64 モノクロ）をタイル状に敷いた背景**（青/黒の模様）
- 中央付近に **32×32 ドットのキャラクター**（`assets/character.png` 由来。 赤/シアン/白の 3 色）
- **テンキー 8/2/4/6** で上下左右に連続移動。 一度押した方向に進み続け、 別の方向キーで向きを変え、 それ以外のキーで停止
- 歩くと **足が動くアニメーション**（4 コマ）
- **BREAK キー** で、 今向いている方向へ **シアンのボール** を投げる（**最大 3 個まで同時**に飛ばせる）
- 画面上部にシアンの **SCORE** 表示
- **PSG サウンド**：単音 BGM が流れ、 発射時はノイズの発射音、 歩くと足音が鳴る
- キャラの透明部分には背景のタイル模様が透けて見え、 移動しても背景は壊れない
- 画面端では止まる

![実行画面: タイル背景の上をキャラが動き、 シアンの SCORE とボールが表示される](img/screenshot.png)

この「背景の上をキャラがアニメしながら動く」 という土台があれば、 アクションゲームの骨格はほぼ揃っています。 ここから敵・スコア・当たり判定などを足していくイメージです。

## 1. FM-7 というマシンを知る

ゲームを作る前に、 FM-7 が「普通の PC とどう違うか」 を 3 つだけ押さえます。 ここが分かると、 後の「なぜこんな回りくどい描き方をするの?」 が腑に落ちます。

### 1-1. CPU が 2 つある（メイン と サブ）

![メイン6809／サブ6809 のデュアルCPU構成](img/arch.svg)

FM-7 には **6809 という CPU が 2 個** 載っています。

```
┌─────────────┐        共有RAM         ┌─────────────┐
│  メイン CPU  │ ←───（小さな郵便受け）──→ │  サブ CPU    │
│ (あなたの    │   $FC80-$FCFF /        │ (画面まわり  │
│  ゲーム本体) │    $D380-$D3FF         │  担当)       │
└─────────────┘                        └──────┬──────┘
                                               │
                                          ┌────▼─────┐
                                          │  VRAM    │ ← 画面
                                          │ (画面メモリ)│
                                          └──────────┘
```

- **メイン CPU**: あなたのゲームのロジック（キャラの座標計算、 当たり判定、 スコア…）を動かす。 このテンプレでは **C 言語** で書く部分。
- **サブ CPU**: 画面（VRAM）とキーボードを担当する。 **VRAM はサブ CPU 側にしか繋がっていない** ので、 メイン CPU は画面に直接絵を描けません。

> **ここが FM-7 の最大の特徴であり、 難所です。** 「絵を描きたいメイン CPU」 と「VRAM を持つサブ CPU」 が別々なので、 メインはサブに**「ここにこう描いて」 とお願いする**しかありません。 そのお願いの仕組みがこのテンプレの肝です（[6 章](#6-画面に絵が出る仕組み核心)）。

### 1-2. 画面は「3 枚の白黒レイヤー」 でできている（plane）

FM-7 の画面は 640×200 ドット。 色は **B（青）・R（赤）・G（緑）の 3 枚の白黒レイヤー（plane）** を重ねて作ります。

```
あるドットの色 = (Bレイヤーのそのドット) + (Rレイヤー) + (Gレイヤー)

  B  R  G   →  色
  0  0  0   →  黒
  1  0  0   →  青
  0  1  0   →  赤
  1  1  0   →  マゼンタ(赤紫)
  0  0  1   →  緑
  1  0  1   →  水色(シアン)
  0  1  1   →  黄
  1  1  1   →  白
```

つまり使える色は **8 色** です。 「青と緑を両方 ON にすると水色」 のように、 光の三原色の足し算で色が決まります。

各 plane は「1 ドット = 1 ビット」 で、 横 8 ドットが 1 バイトにまとまっています（左のドットが上位ビット）。 1 ライン = 640 ドット ÷ 8 = **80 バイト**。

> **FM-7 の描画はこの「3 plane 8 色」 か「モノクロ 1 plane」 が基本**です。 そして **「透過（透明）」 はハードの機能ではありません**。 後で出てくるキャラの「透明部分に背景が透ける」 のは、 FM-7 がやってくれるのではなく、 こちらで「不透明なドットだけ画面に書く」 計算を書いて実現しています（6-3 節）。

> **このテンプレの工夫（plane の役割分担）**: 本テンプレは 3 plane を「**背景 = B plane だけ / 前景（キャラ・弾・文字）= R・G plane だけ**」 と分担して使います。 さらに**パレット**（`$FD38`〜`$FD3F`、 論理色番号→物理色の再割当）で色番号 1=青（背景）、 2・3=赤、 4・5=シアン、 6・7=白 と割り当てます。 これは「2・3 をどちらも赤…」 と二重にしておくことで、 **背景 B の ON/OFF に関係なく R・G だけで前景色（赤/シアン/白）が決まり、 R=G=0 が自動的に背景透過になる**（= 透過用の特別なデータが要らなくなる）ための工夫です。 前景の色番号は **bit0=R, bit1=G**（0=透明 / 1=赤 / 2=シアン / 3=白）になります。

```
VRAM 上のアドレス（サブ CPU から見た番地）:
  B plane: $0000 ～ $3FFF
  R plane: $4000 ～ $7FFF
  G plane: $8000 ～ $BFFF

あるドット (x ドット, y ライン) の入っているバイト:
  byte_col = x / 8          （0〜79）
  addr     = y * 80 + byte_col
  B plane の値 = VRAM[$0000 + addr]
  R plane の値 = VRAM[$4000 + addr]
  G plane の値 = VRAM[$8000 + addr]
```

> 詳しくは [SPRITE.md](SPRITE.md)（sprite データの作り方）と [SUBPROGRAM.md](SUBPROGRAM.md)（描画の実装）にあります。

### 1-3. メインとサブの「郵便受け」 = 共有 RAM

メイン CPU とサブ CPU は、 **共有 RAM** という小さな共通メモリ（128 バイト）で会話します。

- メインから見た番地: `$FC80 - $FCFF`
- サブから見た番地: `$D380 - $D3FF`（同じメモリを別の住所から見ているだけ）

メインが「やりたいこと」 をここに書き、 サブがそれを読んで実行します。 ただし**ルールがあります**:

> **共有 RAM にメインが書き込めるのは「サブを HALT（一時停止）させている間」 だけ。** サブが動いている間に書いても無視されます（FM-7 の仕様）。 このテンプレの関数は内部で自動的に HALT してくれるので、 普段は意識しなくて OK です。 詳細は [SUBPROGRAM.md](SUBPROGRAM.md) §1。

## 2. 開発の全体像

### 2-1. C と アセンブラを併用する

このテンプレは **2 つの言語** を使います。

| 言語 | ツール | 何を書くか |
|---|---|---|
| **C** | CMOC | ゲームのロジック（メイン CPU 側）。 普段あなたが触るのはここ |
| **アセンブラ** | LWASM / LWLINK | サブ CPU 上で動く描画プログラムや、 起動まわりの低レベル処理 |

最初は **C 部分だけ触れば十分** です。 アセンブラ部分（`*.s`）は「すでに用意された部品」 として使い、 慣れてきたら[8 章](#8-もっと作る機能追加レシピ)で手を入れます。

### 2-2. ビルドから起動までの流れ

![ビルドパイプライン](img/build.svg)

`make` 1 回で、 ソースから「ディスクイメージ（D77）」 が出来上がります。

```
   あなたの C / ASM ソース
          │  make（CMOC + LWASM + LWLINK）
          ▼
   本体プログラム (build/hello.bin)
          │  + IPL（起動用の小さなプログラム）
          ▼
   ディスクイメージ (build/hello.d77)
          │  ドライブ 0 にセットして電源 ON
          ▼
   内蔵ブート ROM → IPL → あなたの main() が動く
```

実機がなくても、 FM-7 のエミュレータに D77 を読ませれば動作確認できます。

### 2-3. プログラムは「2 層」 になっている

ここが他のレトロ機と違うので、 最初に頭に入れておくと混乱しません。

```
【メイン CPU 側】= C 言語                【サブ CPU 側】= アセンブラ
┌───────────────────────────┐         ┌──────────────────────────┐
│ c_main.c                  │         │ asm_subprog.s            │
│   ゲームのメインループ     │  お願い  │   VRAM に実際に絵を描く   │
│   キー入力を読む          │ ──────→ │   コマンドを 1 つ実行して  │
│   座標を計算する          │ (共有RAM) │   返事する                │
│   「ここに描いて」 と頼む  │         │                          │
└───────────────────────────┘         └──────────────────────────┘
```

- **C 側**（`c_main.c`, `c_subprog.c`, `c_subsys.c`）: あなたのゲーム。
- **サブ側プログラム**（`asm_subprog.s`）: 起動時にサブ CPU の RAM へ送り込まれ、 そこで動く「描画エンジン」。 メインからの命令（コマンド番号）を受けて VRAM を書きます。

C 側は `sub_blit_sprite(...)` のような関数を呼ぶだけ。 その関数が裏で「サブにコマンドを送る」 仕事をします。

## 4. はじめてのビルドと実行

プロジェクトのルート（`Makefile` がある場所）で:

```bash
make
```

成功すると `build/hello.d77` が出来ます。 これをエミュレータのドライブ 0 にセットして起動すると、 [0 章](#0-はじめに--完成形と読み方)の画面が出ます。

うまくいかないとき:

- `cmoc: command not found` → CMOC が PATH に無い。 [README.md](../README.md) の環境構築へ

> **クリーンビルド**: 変な状態になったら `make clean && make` でやり直せます。

## 5. プロジェクトの中身を見る

### 5-1. ファイル構成

```
FM7BaseCode/
├── config.mk          ← プロジェクト設定（名前・配置アドレス）。 まずここ
├── Makefile           ← ビルド手順（基本いじらない）
├── README.md          ← 環境構築・ビルド・実行
├── docs/              ← ドキュメント（このファイルもここ）
├── assets/
│   ├── character.png  ← キャラ画像（差し替え可能）
│   ├── sprite_to_asm.py  ← PNG → sprite データ変換
│   └── （font 関連）
├── src/
│   ├── c_main.c       ← ★ゲーム本体。 普段触るのはここ
│   ├── c_subprog.c/.h ← サブへの描画依頼 API（C 側の窓口）
│   ├── c_subsys.c/.h  ← サブシステム ROM 経由の API（文字表示など）
│   ├── asm_subprog.s  ← ★サブ CPU 上で動く描画エンジン
│   ├── asm_test.s ← サブへコード/データを送り込む低レベル処理
│   ├── asm_kbd.s      ← キーボード入力
│   ├── asm_ipl.s      ← 起動ローダ
│   ├── asm_crt0.s     ← C の起動準備
│   └── （その他 ASM）
└── build/             ← make の生成物（.bin / .d77 など）
```

最初に意識するファイルは **`src/c_main.c`**（あなたのゲーム）と **`assets/character.png`**（キャラ絵）の 2 つだけで十分です。

### 5-2. 起動の流れ

![起動シーケンス](img/boot.svg)

![メモリマップ](img/memmap.svg)

電源 ON から `main()` が動くまで:

```
電源ON → 内蔵ブートROM → ディスクの IPL を読んで実行
       → IPL が本体プログラムをメモリに展開して JMP
       → asm_crt0.s（スタック準備など）→ あなたの main()
```

> FM-7 は通常 **BASIC モード (BASIC モードのブート ROM)** で起動しますが、 BASIC のブート ROM でも FDD にディスクが入っていれば、 その先頭セクタ (IPL) を自動で読み込んで起動します。 だから「ディスクを入れて電源 ON」 だけでゲームが立ち上がります (= 特別な DOS モードは不要)。

`main()` に来た後の流れは `c_main.c` を読めば分かります。 起動の低レベルな詳細は [GAMESUB.md](GAMESUB.md) と [DETAIL.md](DETAIL.md) に。

### 5-3. `config.mk` — 唯一いじる設定ファイル

```makefile
NAME = hello        # 生成物の名前（hello.d77 など）と ディスクラベル
ORG  = 0x0400       # 本体プログラムをメモリのどこに置くか (既定 $0400)
```

普段は触りませんが、 プロジェクト名を変えたいときは `NAME` を変えるだけ。 Makefile が追従します。

## 6. 画面に絵が出る仕組み（核心）

ここがこのテンプレの心臓部です。 「メイン CPU が VRAM を直接触れない」 という制約を、 どう乗り越えているか。

### 6-1. サブ CPU に「描画エンジン」 を送り込む

![サブシステム側 メモリマップ](img/submap.svg)

起動時、 メインは `asm_subprog.s` をコンパイルした小さな機械語を **サブ CPU の RAM（`$C300`）に転送** し、 そこで実行させます。 これが「描画エンジン」 で、 以後ずっとサブ CPU 上に常駐します。 VRAM が `$0000–$BFFF` の 48 KB を占めるため、 自由に使える RAM は `$C000–$CFFF` の 4 KB だけ。 ここにコード・sprite・背景タイルを上図のように詰め込んでいます。

この「サブにコードを送り込む」 仕掛けが **TEST コマンド**（サブシステム ROM の拡張機能）です。 仕組みの詳細は [SUBPROGRAM.md](SUBPROGRAM.md) §1-2。 初心者は「メインがサブにプログラムを送って動かす魔法」 とだけ理解すれば OK。

### 6-2. メインは「コマンド番号」 で描画を頼む

描画エンジンは、 共有 RAM に置かれた **コマンド番号** を見て、 対応する処理を 1 つ実行します。

| 番号 | 名前 | やること |
|---|---|---|
| `$00` | NOP | 何もしない |
| `$01` | PUT_CELL | 8×8 ドットを 1 色で塗る |
| `$02` | CLR_CELL | 8×8 ドットを黒で消す |
| `$03` | CLS | 画面全体を黒で消す |
| `$04` | BLIT_SPRITE | 32×32 のキャラを描く |
| `$05` | ERASE_BOX | 32×32 の領域を背景で塗り直す |
| `$06` | MOVE_SPRITE | キャラを旧位置→新位置へ移動（消去+描画を一気に） |
| `$07` | DRAW_BG | 背景タイル（backimage.png 64×64）を全画面に敷く |
| `$08` | DRAW_CHAR | 8×8 の文字を 1 つ描く（テキスト表示の部品） |
| `$09` | DRAW_BALL | 8×8 のシアンの丸（ボール）を描く |
| `$0A` | ERASE_BALL | 8×8 のボールを消す（R/G を 0 クリア、 背景は残る） |

C 側では、 これらに対応する関数が用意されています（`c_subprog.h`）:

```c
sub_load_bgtile();                      // 背景タイル(64x64)を sub へ転送
sub_draw_bg();                          // 背景タイルを全画面に敷く
sub_blit_sprite(id, x, y);              // キャラを描く
sub_move_sprite(id, ox, oy, nx, ny);    // キャラを移動
sub_erase_box(x, y);                    // キャラを消す（背景に戻す）
sub_put_cell(x, y, color);              // 8x8 を 1 色で塗る
sub_draw_text(col, row, "SCORE", 2);    // 文字列を表示（シアン=2）
sub_draw_ball(x_byte, y_line, color);   // 8x8 のシアンの丸を描く（シアン=2）
sub_erase_ball(x_byte, y_line);         // ボールを消す（R/G を 0 クリア、 背景は残る）
```

あなたが C で `sub_blit_sprite(0, 30, 80)` と書くと、 関数の中で「コマンド `$04` + 引数を共有 RAM に書く → サブに実行を頼む」 という処理が走り、 サブ CPU が VRAM にキャラを描きます。

### 6-3. キャラ（sprite）のデータ構造

[1-2 節](#1-2-画面は3枚の白黒レイヤーでできているplane)で述べたとおり、 このテンプレは **背景 = B plane / 前景（キャラ）= R・G plane** と分担しています。 だからキャラのデータは **R plane と G plane の 2 枚だけ**で済みます。

```
1 sprite = 256 バイト
┌────────────────────┐
│ R plane  (128 byte) │ 赤成分のドット  ┐ この 2 枚だけ画面（VRAM）の
│ G plane  (128 byte) │ 緑成分のドット  ┘ R/G plane に転送する
└────────────────────┘
   R=0,G=0 → 透明（背景が透ける） / R=1 → 赤 / G=1 → シアン / R+G → 白
```

**透過には専用の mask も計算も要りません。** 「R も G も 0 のドット = 透明」 と決めておき、 キャラの R/G をそのまま画面の R/G plane に**上書き (store)** するだけです:

```
画面_R = キャラ_R
画面_G = キャラ_G        （B plane = 背景 には一切触らない）
```

透明ドット（R=G=0）は画面の R/G に 0 が入りますが、 **背景は別の B plane に分けてある**ので（描画で触らない）、 そこは背景の B = 青がそのまま透けます。 セルを毎回まるごと上書きするので前のキャラの残像も自動で消え、 AND/OR のような読み書き合成 (RMW) は要りません。 この処理を書いているのがサブ側プログラム（`asm_subprog.s`）です。

> キャラの絵（`character.png`）は **赤/シアン/白の 3 色**に量子化されます（中間色は隣のドットへ誤差を散らす「ディザ」 で、 赤シアン/赤白/シアン白の混色として 6 色相当に見せます）。 データの作り方・配置の詳細は [SPRITE.md](SPRITE.md)。

### 6-4. 動かしても背景が壊れない仕組み

キャラを動かすとき、 旧位置のキャラを消さないと残像が残ります。 このテンプレは「旧位置の **R/G plane を 0 でクリアする**」 だけです（`MOVE_SPRITE`）:

```
move のとき:
  1. 旧位置の R/G を 0 クリア（= 前景を消す。 背景の B = 青 は残るので背景が戻る）
  2. 新位置にキャラを R/G へ上書き (store) 描画（透明部分は R/G=0 → B 背景が透ける）
```

背景（B plane）には一切触らないので、 「背景を保存して戻す」 ような退避バッファは不要で、 残像も出ません。

## 7. 改造してみよう（ハンズオン）

ここからは手を動かします。 どれも **`src/c_main.c` を編集 → `make` → エミュで確認** の繰り返しです。

### 7-1. キャラの初期位置を変える

`c_main.c` の `main()` の最初:

```c
unsigned char x       = 30;   // ← 横位置（0〜76、 8ドット単位）
unsigned char y       = 80;   // ← 縦位置（0〜168、 1ライン単位）
```

`x = 5; y = 5;` にすれば左上に、 `x = 70; y = 150;` にすれば右下に出ます。

> **なぜ x は 0〜76?** キャラは横 32 ドット = 4 バイト幅。 画面は 80 バイト幅なので、 左端を置けるのは 80 - 4 = 76 まで。 x は「8 ドット単位（バイト単位）」 なので 1 増やすと 8 ドット動きます。

### 7-2. 移動速度を変える

```c
#define STEP_X_BYTES        1   // 横移動：1 = 8ドット/回
#define STEP_Y_LINES        4   // 縦移動：4 ライン/回
```

`STEP_Y_LINES` を `8` にすると縦移動が倍速に。 横は 1 バイト（8 ドット）が最小単位なので、 それより細かくしたいときは別の工夫が要ります（[10 章](#10-用語集)「ドット単位移動」 参照）。

### 7-3. 歩行アニメの速さを変える

```c
#define ANIM_PERIOD         3   // 3回移動するごとに足のコマを進める
```

小さくするとパタパタ速く、 大きくするとゆっくり歩きます。 `1` にすると毎回コマが変わって慌ただしく、 `8` だとのんびり。

### 7-4. 背景の色を変える

背景は `assets/backimage.png`（**64×64 ドットのモノクロ画像**）を全画面にタイル状に敷いたものです。 模様を変えるには、 この PNG を描き換えるだけです:

- サイズは **64×64 ピクセル**、 モノクロ（明るいドット = 点が出る = 色番号 1 = 青、 暗いドット = 点なし = 黒）
- 描き換えたら `make`。 `scripts/bgtile_to_asm.py` が PNG を読んでタイルデータ（B plane 512 byte）に変換し、 サブ側 `DRAW_BG` がそれを全画面（横 64px・縦 64 line 周期）に敷きます

> 背景の「色」 そのもの（青）を変えたい場合は、 パレット（[1-2 節](#1-2-画面は3枚の白黒レイヤーでできているplane)）の色番号 1 を別の色にします（`src/asm_kbd.s` の `palette_init`、 `$FD39` に書く値）。 前景（赤=2,3 / シアン=4,5 / 白=6,7）の色番号とぶつからないよう注意してください。

### 7-5. 自分のキャラ画像に差し替える

`assets/character.png` を自作の画像に置き換えるだけです。 ルール:

- サイズは **128×128 ピクセル**
- 4 行 × 4 列 = 16 個の 32×32 キャラが並んだ「スプライトシート」
  - 行 = 向き（上から **下・上・右・左**）
  - 列 = 歩行アニメの 4 コマ
- **背景は透明**（アルファ 0）にする。 透明部分が「背景が透ける部分」 になる
- 色は **赤/シアン/白の 3 色**に自動的に量子化される（中間色は隣のドットへ誤差を散らすディザで、 赤シアン/赤白/シアン白の混色として 6 色相当に見える）。 透明部分（アルファ 0）はそのまま背景が透ける

差し替えたら `make`。 `scripts/sprite_to_asm.py` が PNG を読んでデータに変換します。 変換の詳細ルールは [SPRITE.md](SPRITE.md) §1-4。

### 7-6. 操作キーを変える

```c
#define KEY_UP              '8'
#define KEY_DOWN            '2'
#define KEY_LEFT            '4'
#define KEY_RIGHT           '6'
```

`'8'` を `'W'` にすれば W キーで上、 のように変えられます。 キーは ASCII コードで判定しているので、 文字をそのまま書けます。

### 7-7. BREAK キーでボールを投げる

**BREAK キーを押すと、 キャラが今向いている方向へシアンのボールを投げます。** 向き（`dir_idx`: 下=0 / 上=1 / 右=2 / 左=3）に飛び、 画面端（`bx>79` / `by>192` / 端でこれ以上進めない位置）に達すると消えます。

**ボールの色はシアン（前景 `color 2`）です。** 前景の色番号は bit0=R, bit1=G なので、 `2`（= G のみ）がシアンになります（[1-2 節](#1-2-画面は3枚の白黒レイヤーでできているplane)の前景色の話を参照）。

**速度**は、 縦が 8 ライン／フレーム（`BALL_STEP_Y = 8`）、 横が 16 ドット = 2 バイト／フレーム（`BALL_STEP_X = 2`）です。 横はキャラの横移動の**倍速**にしています。 横方向はキャラと同じ速さだと遅くて見づらいので、 あえて倍にしてあります。

**3 連射（最大 3 個まで同時）**: ボールは同時に最大 3 個まで飛ばせます（`BALL_MAX = 3`）。 BREAK の立ち上がりエッジごとに、 **空きスロットがあれば 1 個**発射します。 すでに 3 個飛んでいるときは発射しません。 前の弾が飛行中でも、 3 個未満なら次を撃てます。

BREAK キーは少し特殊で、 **テンキーなどの普通のキー入力（`$FD01`）には流れてきません**。 代わりにメイン側の `$FD04` の bit1 に出ます（0=押下 / 1=非押下 の active-low）。 そこで `break_check()`（`asm_kbd.s` 実体、 `c_subprog.h` 宣言）がその bit を読み、 押下中なら 1 / 非押下なら 0 を返します。

メインループは毎フレーム `break_check()` を読み、 **「前回 0・今回 1」 の立ち上がりエッジ**のときだけ、 空きスロットがあれば 1 発撃ちます（`c_main.c` の `brk_prev` で前回の値を覚えておき、 押しっぱなしで連射しないようにしています）。

> ボールの色や丸の形を変えたいときは、 サブ側プログラム（`asm_subprog.s`）の丸データ（`BALL_GLYPH`）を編集します。 描画は文字（`DRAW_CHAR`）と同じ仕組みで、 8×8 セルの R/G を上書き (store) し、 丸のドットに色を、 外側に 0 を置きます（外側は R/G=0 → B 背景が透ける）。
>
> **既知の制限**: 前景はすべて R/G への上書き (store) なので、 ボールがキャラや SCORE 文字（これらも前景 = R/G）の上を通ると、 ボールの 8×8 セルがその部分の前景を 0 で潰してしまいます（描画でも消去でも同じ。 背景タイルは B plane なので残ります）。 このサンプルでは許容し、 SCORE は毎フレーム描き直して復帰させています。

### 7-8. フレームレートを安定させる（フレームペーシング）

ここは初心者が必ずつまずくところなので、 しっかり説明します。

**何が問題か**: 何も対策しないと、 ゲームの速さがフレームごとにバラついてしまいます。 キャラが動いている間は重い描画（`MOVE_SPRITE`）を出すのでループ 1 周が遅くなり、 止まっている間は処理が軽いので 1 周が速くなります。 すると「止まっているときだけ弾が速い」 といった、 おかしな挙動になります。

**なぜ VSYNC で同期しないのか**: ふつうのレトロ機なら「画面の描き替えタイミング（VSYNC）に合わせて 1 フレーム待つ」 方法を使います。 しかし **FM-7（初代）では、 メイン CPU から VSYNC を検出する手段がありません**。 サブ側の表示ステータス（`$D430`）や `$FD12` は FM77AV 以降でしか有効ではなく、 初代 FM-7 ではこれらを読んでも素通りしてしまい、 待ちになりません。 つまり「本物の VBlank 同期」 は初代 FM-7 では使えません。

**採用している方法 = メイン CPU のタイマで時間を測る（deadline 方式）**: 実は FM-7 のメイン CPU 自身に **約 2ms 周期のタイマ割り込み（IRQ）** があります。これを時間の基準にします。割り込みのたびに「経過時間カウンタ」を +1 しておき、 メインループでは「カウンタを 0 に戻す（`timer_start()`）→ 処理 → カウンタが `FRAME_TARGET`（例: 16 ≒ 32ms）になるまで待つ（`timer_get()`）」 を繰り返します。動いているフレームでも止まっているフレームでも、 1 周がきっかり `FRAME_TARGET × 約2ms` に揃い、 弾やキャラの速さがバラつきません。

**なぜ「ポーリング」ではなく「割り込みで数える」のか**: 時間のフラグは**1 個のラッチ**で、 読むと消えます。 処理に時間がかかってその間に複数回タイマが過ぎても「1 回」 にしか見えず、 **何回ぶん時間が過ぎたか分かりません**。 だから「経過時間を測って一定値まで待つ」 方式には、 **割り込みのたびにカウンタを +1 する**必要があります（割り込みなら処理中も数え続けられる）。

**BREAK と干渉しない（IRQ であって FIRQ ではない）**: 割り込みには IRQ と FIRQ の 2 種類があり、 BREAK キー（8-3）は **FIRQ** 要因です。もし FIRQ を使うと、 BREAK を押している間ずっと割り込みが止まらずゲームが固まります（実際に起きました）。メインのタイマは **IRQ** 要因なので、 **IRQ だけ許可して FIRQ は止めたまま**にすれば BREAK と干渉しません（BREAK は `break_check()` がポーリングで読む）。なお、 キーボード入力もこの IRQ で読みます（[TIMER.md §6](TIMER.md)）。

**速さの調整**: `FRAME_TARGET` を増やすと 1 フレームが長く（全体がゆっくりに）なります（1 tick ≈ 2ms。 16 ≒ 32ms/30fps）。仕組みの詳細は [TIMER.md](TIMER.md) を参照。

**おまけ（デバッグ表示）**: 待ち中に空回りした回数を `SCORE` 欄に数字で出しています。 安定した値なら順調、 上限付近に張り付くなら **タイマ割り込みが来ていない**サインです（[TIMER.md](TIMER.md) の検証ポイント参照）。

## 8. もっと作る（機能追加レシピ）

ここからは「新しいことをやる」 段階。 アセンブラにも少し触れます。

### 8-1. 新しい描画コマンドを追加する（例：矩形を塗る）

「画面の好きな場所に、 好きな色の四角を描く」 コマンドを追加してみます。 描画は **サブ側**（`asm_subprog.s`）に処理を足し、 **C 側**（`c_subprog.c/.h`）に呼び出し口を作り、 **`c_main.c`** から呼びます。 3 ステップです。

実は既に近いもの（`PUT_CELL` = 8×8 を 1 色で塗る、 コマンド `$01`）があるので、 まずはそれを使ってみるのが早道です。

**ステップ 1: C ヘッダにすでにある関数を確認**（`c_subprog.h`）

```c
void sub_put_cell(unsigned char x_byte, unsigned char y_line,
                  unsigned char color);
```

**ステップ 2: c_main.c から呼ぶ**

```c
// 起動シーケンスの後、 メインループの前あたりに:
sub_put_cell(10, 20, 1);   // (横10バイト, 縦20ライン) に 赤(1) の 8x8 を描く
sub_put_cell(11, 20, 2);   // 隣に シアン(2)
```

`make` して確認。 色番号は[1-2 節の表](#1-2-画面は3枚の白黒レイヤーでできているplane)（0=黒〜7=白）。

**自分で全く新しいコマンドを作る場合**（例：横一列をまとめて塗る `$08 DRAW_LINE`）は、 次の 3 箇所に手を入れます:

1. `asm_subprog.s` の入口（コマンド処理）に分岐を追加し、 処理本体（handler）を書く
   ```asm
   ; 入口の cmpa の並びに追加:
                   cmpa    #$08
                   lbeq    do_draw_line
   ; どこかに handler を追加（末尾で clr CMD_REG するのを忘れずに）:
   do_draw_line:
                   ; ... 共有 RAM のパラメータ（PARAM=$D394〜）を読んで描く ...
                   clr     CMD_REG     ; ★必須（理由は 9 章 #4）
                   rts
   ```
2. `c_subprog.h` にコマンド番号と関数プロトタイプ、 `c_subprog.c` に関数本体（`sub_halt()` → 共有 RAM にパラメータ → `sub_call(SUB_PROG_ADDR)` の定型）を追加
3. `c_main.c` から新関数を呼ぶ

> handler の書き方（VRAM アドレス計算、 plane への書き込み）は `asm_subprog.s` の既存 handler（`do_put_cell` など）が手本になります。 詳細は [SUBPROGRAM.md](SUBPROGRAM.md) §3。
>
> **注意**: サブ側に処理を足すとプログラムが長くなり、 sprite データの置き場所とぶつかることがあります（[9 章](#9-ハマりどころ必読) #2）。 `make` 後にサイズ警告が無いか確認を。

### 8-2. キャラを 2 体出す

メインループの変数（`x`, `y`, `dir` など）を**配列**にして、 各キャラを順番に動かします。 描画コマンドは 1 つずつ順番に発行されるので、 ループで複数体を `sub_blit_sprite` / `sub_move_sprite` すれば OK。

```c
unsigned char ex = 50, ey = 100;   // 敵の座標
// ... 敵の移動ロジック ...
sub_move_sprite(enemy_frame, old_ex, old_ey, ex, ey);
```

注意点として、 キャラ同士が重なると「後から描いた方」 が上に来ます。 また 2 体が近いと、 片方を消す `MOVE_SPRITE` がもう片方を背景で塗ってしまうことがあるので、 重なり処理は工夫が要ります。

### 8-3. 当たり判定（AABB）

キャラはどれも 32×32 の四角なので、 2 つの四角が重なっているかで判定できます（AABB = 軸並行境界ボックス）:

```c
// a と b が重なっているか（座標は左上、 サイズ 32x32 と仮定）
// x はバイト単位なので 8 倍してドットに、 y はそのままライン
int ax = a_x * 8, ay = a_y;
int bx = b_x * 8, by = b_y;
if (ax < bx + 32 && ax + 32 > bx &&
    ay < by + 32 && ay + 32 > by) {
    // 当たっている！
}
```

### 8-4. スコアなどの文字を出す

**`sub_draw_text` を使うだけ** です。 同梱の 8×8 フォントで、 指定位置に文字列を描きます:

```c
sub_draw_text(2, 4, "SCORE 0000", 7);   // (col 2, row 4) に 白(7) で表示
```

- `col`: 横位置（0〜79、 1 文字 = 8 ドット = 1 単位）
- `row`: 縦位置（line 単位。 文字の高さは 8 ライン）
- 第 3 引数: ASCII 文字列（大文字・数字・記号。 内蔵フォントは Press Start 2P 由来）
- 第 4 引数: 色（0〜7、 [1-2 節の表](#1-2-画面は3枚の白黒レイヤーでできているplane)）

文字の**隙間は背景が透ける**ので、 背景タイルの上に置いても自然です。 スコアを更新したいときは、 数値を文字列に変換して毎回 `sub_draw_text` で描き直します（同じ位置に上書き）。

```c
// 例: スコアを 4 桁で表示する簡単な実装
char buf[5];
buf[0] = '0' + (score / 1000) % 10;
buf[1] = '0' + (score / 100)  % 10;
buf[2] = '0' + (score / 10)   % 10;
buf[3] = '0' + (score)        % 10;
buf[4] = 0;
sub_draw_text(8, 4, buf, 2);   // "SCORE " の右隣に数字だけ更新（シアン=2）
```

> **HUD とキャラの重なり注意**: テキスト（HUD）もキャラも前景（R/G plane）なので、 テキストの上をキャラが通ると、 キャラ移動の旧位置消去（R/G クリア）でテキストも消えてしまいます。 スコア表示は画面上部など、 キャラが通らない場所に置くのが定石です。 どうしても重なる場合は、 キャラ移動のたびに HUD を描き直す工夫が要ります（このテンプレの `c_main.c` は移動後に毎回 SCORE を描き直しています）。
>
> 内部の仕組み（`DRAW_CHAR` コマンド = glyph の立ったドットだけ指定色で描き、 透明部は背景を残す）は [SUBPROGRAM.md](SUBPROGRAM.md) §3、 フォントデータは [FONT.md](FONT.md) を参照。

### 8-5. 音を鳴らす（PSG）

このテンプレには **PSG（AY-3-8910）サウンドが実装済み**です。単音 BGM が流れ、 BREAK の発射音（ノイズ）と歩行音が鳴ります。PSG は **メイン CPU 側の I/O**（`$FD0D`/`$FD0E`）なので、 グラフィックのサブシステムとは独立に、 `c_main` 側だけで鳴らせます。

- **ch A=発射音（ノイズ）/ ch B=歩行音 / ch C=単音 BGM** の 3 チャンネル固定（競合なし）。
- メインループ先頭で毎フレーム `sound_tick()` を呼んで BGM を進め、 効果音を減衰させます（フレームペーシングの 20ms ティックで駆動）。発射時に `sfx_ball()`、 1 歩ごとに `sfx_walk()`。
- 曲を変えたいときは `src/c_sound.c` の `bgm_period[]` / `bgm_frames[]` を書き換えるだけ。

仕組み・チャンネル割り当て・BGM データ形式・効果音の作り方の詳細は **[SOUND.md](SOUND.md)** を参照。

## 9. ハマりどころ（必読）

サブシステム周りで初心者が特に踏みやすい点を 6 つ挙げます。 サブシステムプログラミング全般の疑問・注意点 (スクロールオフセット、 VRAM 非表示領域、 ワーク領域、 HALT 等) は [SUBPROGRAM.md](SUBPROGRAM.md) §9 (= よくある疑問と注意点) も参照してください。

### #1 画面の左上に変な点が残る／起動メッセージが消えない

サブシステム ROM が起動時に**カーソルを点滅**させて VRAM に書き込んでいます。 対策は起動時に「カーソル OFF」 と「画面クリア」 を行うこと。 テンプレの起動シーケンス（`c_main.c` の `subsys_call(SCMD_CURSOR, ...)` → `sub_draw_bg()`）が既にやっています。 これを消すと点が残ります。

### #2 機能を足したら画面が乱れる／キャラが化ける

サブ側プログラム（`asm_subprog.s`）にコマンドを足すとプログラムが長くなり、 **sprite データの置き場所とぶつかる**ことがあります。 プログラムが sprite データ領域を侵食すると、 データを命令として実行して暴走します。

実際このテンプレでも、 ボール機能（`DRAW_BALL` / `ERASE_BALL`）を足したことで subprog コードが 838 バイトに増え、 sprite の置き場所を `$C600` から **`$C700`** へ繰り上げました。 現在は subprog コード = `$C300-$C6FF`、 sprite データ = `$C700-$CAFF` です（`SPRITE_BASE` と `SUB_CHARS_ADDR` はどちらも `$C700`）。

対策: `make` 後、 `build/subprog.bin` のサイズを見て、 `$C300 + サイズ` が `$C700`（`SPRITE_BASE`）を超えていないか確認。 超えそうなら `SPRITE_BASE`（`asm_subprog.s`）と `SUB_CHARS_ADDR`（`c_subprog.h`）を両方とも後ろにずらす。

### #3 ヘッダを書き換えたのに反映されない

`c_subprog.h` などのヘッダを変えても再ビルドされない場合は `make clean && make`。 通常はヘッダ依存が効くようにしてありますが、 念のため。

### #4 描画が二重になる／おかしい

サブ側の各 handler は、 処理の最後に必ず `clr CMD_REG`（コマンド番号を 0 に戻す）を書きます。 これを忘れると、 同じコマンドが 2 回実行されることがあります。 新しい handler を書いたら必ず付けてください。

### #5 壁にぶつかるとキャラが残る／移動で背景が汚れる

「キャラの下の背景を保存→復元」 方式だと、 既にキャラが描かれた画面を背景として保存してしまい、 残像が出ます。 このテンプレは「移動時に旧位置を**背景パターンで計算し直して塗る**」 方式でこれを回避しています。 自前で背景処理を変えるときはこの点に注意。

### #6 速さがフレームごとにバラつく／停止中だけ弾が速い

**VSYNC は初代 FM-7 ではメイン CPU から取れません**（`$D430` や `$FD12` は FM77AV 以降専用）。 そのため VSYNC 同期に頼らず、 メイン CPU の約2ms周期タイマ IRQ を数える経過時間カウンタを使い、 一定 tick まで待つ**フレームペーシング**（deadline 方式）で速度を安定させています（[7-8 節](#7-8-フレームレートを安定させるフレームペーシング) / [TIMER.md](TIMER.md)）。 速さは `FRAME_TARGET` で調整します。

## 10. 用語集

| 用語 | 意味 |
|---|---|
| **メイン CPU / サブ CPU** | FM-7 の 2 つの 6809。 メイン=ゲーム本体、 サブ=画面・キー担当 |
| **VRAM** | 画面メモリ。 サブ CPU 側にある |
| **plane（プレーン）** | 色の 1 成分（B/R/G）を表す白黒レイヤー。 3 枚重ねて 8 色。 本テンプレは背景=B / 前景=R・G と分担 |
| **共有 RAM** | メインとサブが会話する小さな共通メモリ（`$FC80` / `$D380`） |
| **HALT** | サブ CPU を一時停止させること。 共有 RAM への書き込みに必要 |
| **TEST コマンド** | サブにコード/データを送り込むためのサブシステム ROM の拡張機能 |
| **subprog** | サブ CPU 上に常駐する自前の描画エンジン（`asm_subprog.s`） |
| **sprite** | 動く小さな絵。 ここでは 32×32 ドット。 前景なので R/G の 2 plane（256 byte）で持つ（赤/シアン/白） |
| **mask** | 不透明ドットを表す概念。 このテンプレでは専用データも mask 計算も持たず、 R/G への上書き (store) と背景の B plane 分離だけで透過する（R=G=0 が透明） |
| **blit** | 絵をメモリ（VRAM）に転送・描画すること |
| **パレット** | 論理色番号→物理色の対応表（`$FD38`〜`$FD3F`）。 本テンプレは 1=青, 2,3=赤, 4,5=シアン, 6,7=白 |
| **背景タイル** | `backimage.png`（64×64 モノクロ）を全画面に敷いた背景。 B plane のみ使う |
| **IPL** | ディスクの先頭にある起動ローダ |
| **CMOC / LWASM** | C コンパイラ / アセンブラ |
| **D77** | FM-7 のディスクイメージ形式 |
| **byte 単位 / line 単位** | 横はバイト（8ドット）単位、 縦はライン（1ドット）単位で扱う |
| **ドット単位移動** | 横を 8 ドット未満で動かす技法。 plane のビットシフトが必要（応用） |
| **BREAK キー** | 普通のキー入力（`$FD01`）に流れず、 メイン側 `$FD04` の bit1（0=押下の active-low）に出る特殊キー。 `break_check()` で読む |
| **エッジ検出（立ち上がり）** | 「前回 0・今回 1」 の瞬間だけ反応させる手法。 押しっぱなしでの連射を防ぎ、 1 回押す＝1 発にする |
| **VSYNC** | 画面の描き替えタイミング。 初代 FM-7 ではメイン CPU から検出できない（`$D430` / `$FD12` は FM77AV 以降のみ有効） |
| **フレームペーシング** | メイン CPU の約2ms周期タイマ IRQ を数える経過カウンタを使い、 ループ 1 周を `FRAME_TARGET × 約2ms` に揃えて速さを安定させる手法（deadline 方式） |
